01The big picture

What actually changed, and why it matters.

Before, tasks.service.ts held a plain array (private tasks: Task[]) and a counter for ids. It worked — until the process restarted, at which point every task vanished. A real app needs data that outlives the process, can be queried efficiently, and whose schema is version-controlled. That's the job of a database plus an ORM (Object–Relational Mapper).

Before — in-memory

private tasks: Task[] = [];
private nextId = 1;

create(dto) {
  const task = { id: this.nextId++, ...dto };
  this.tasks.push(task);
  return task; // gone on restart
}

After — Postgres via repository

constructor(
  @InjectRepository(Task)
  private repo: Repository<Task>,
) {}

async create(dto) {
  const task = this.repo.create(dto);
  return this.repo.save(task); // INSERT
}

The three things the task asked for, each delivered:

GoalHowWhere
EntitiesA class decorated with @Entity/@Column that both describes the table and is the TypeScript type.tasks/entities/task.entity.ts
Repositories injected into the service@InjectRepository(Task) hands the service a Repository<Task> via DI.tasks/tasks.service.ts
MigrationsVersioned SQL files generated by diffing entities against the DB; applied deliberately with a CLI script.database/migrations/*.ts
✓ Verified, not assumed

Every claim here was checked against a live Postgres (started with Docker): the app boots and connects, full CRUD works, filtering + search work, and — the whole point — data survives a restart. We restarted the server three times and the same row (id=3 "Persisted task") was still there each time.

The new link in the chain is the last two boxes. The service no longer manipulates an array — it asks a repository to talk to Postgres and awaits the result. Because that's a network round-trip, every service method became async, which (as we'll see) has consequences upstream.

03The entity = table + type

src/tasks/entities/task.entity.ts

One class now plays two roles. To TypeScript it's the shape of a task; to TypeORM the decorators are the schema. TypeORM reads them at startup to map each property to a column.

@Entity('tasks')              // backed by a table named "tasks"
export class Task {
  @PrimaryGeneratedColumn()      // auto-increment PK (Postgres SERIAL)
  id: number;

  @Column()
  title: string;

  @Column({ type: 'text', nullable: true })   // matches the optional ?
  description?: string;

  @Column({ type: 'enum', enum: TaskStatus, default: TaskStatus.OPEN })
  status: TaskStatus;

  @CreateDateColumn()           // set automatically on INSERT
  createdAt: Date;
}
Why an enum column

Declaring type: 'enum' makes Postgres create a real ENUM type, so the database itself rejects an invalid status — defense in depth on top of the @IsEnum DTO validation. You can see it in the generated migration: CREATE TYPE "tasks_status_enum" AS ENUM('OPEN','IN_PROGRESS','DONE').

Subtle: who applies the default?

It's tempting to think repo.create({title}) fills in status: 'OPEN' in memory. It does not — we confirmed this by reading TypeORM's source: create() only does new Task(). The OPEN value is supplied by the Postgres column DEFAULT on INSERT and read back via the RETURNING clause that save() uses. The code comment was corrected to say exactly this.

04One connection, two consumers

src/database/data-source.ts — the single source of truth.

There are two completely separate programs that need to know how to reach the database: the running app and the migration CLI (which runs outside NestJS). If each defined its own connection, they could drift apart. So we define the options once and both import them.

import 'dotenv/config';          // the CLI runs outside Nest, so it needs dotenv
import { DataSource, DataSourceOptions } from 'typeorm';
import { Task } from '../tasks/entities/task.entity';

export const dataSourceOptions: DataSourceOptions = {
  type: 'postgres',
  host: process.env.DB_HOST ?? 'localhost',
  port: parseInt(process.env.DB_PORT ?? '5432', 10),
  username: process.env.DB_USERNAME ?? 'postgres',
  password: process.env.DB_PASSWORD ?? 'postgres',
  database: process.env.DB_NAME ?? 'tasks',
  entities: [Task],                              // explicit = robust src/ vs dist/
  migrations: [__dirname + '/migrations/*.{ts,js}'], // glob: they accumulate
  synchronize: false,                            // ← critical, see below
};

export default new DataSource(dataSourceOptions); // the CLI loads this default export
Why forRoot, not forRootAsync+ConfigService

The usual advice is to build options from ConfigService. But the CLI can't use Nest's DI, so the options can't depend on ConfigService if they're to be shared. Reaching for forRootAsync here would force a second copy of the config — exactly the drift we're avoiding. Sharing one plain object is the better call for this project.

Why dotenv appears twice (and it's fine)

The app loads env via ConfigModule; this file also does import 'dotenv/config'. No conflict: dotenv never overwrites a key that's already set, and both read the same .env, so the values are identical no matter which runs first. The CLI path uses only dotenv.

⚠ The synchronize footgun

synchronize: true makes TypeORM auto-alter your tables to match the entities on every boot — and it will silently drop columns and data to do it. It's seductive in a tutorial and catastrophic in production. We keep it false and use migrations, which is the whole point of this step.

05Module wiring: forRoot vs forFeature

Two registrations with two different jobs.

CallWhereWhat it does
TypeOrmModule.forRoot(dataSourceOptions)app.module.tsOpens the one connection pool for the whole app. Done once, at the root.
TypeOrmModule.forFeature([Task])tasks.module.tsRegisters a Repository<Task> provider in this feature's DI scope so the service can inject it.

Mental model: forRoot = "here's the database." forFeature = "this module works with these tables, give it their repositories." Forget forFeature and you get the classic Nest error "Can't resolve dependencies of TasksService… TaskRepository".

06The service: injecting a repository

src/tasks/tasks.service.ts

The repository is a thin data-access object with find/save/delete. A few patterns worth internalizing:

// LIST — typed WHERE + pagination in one query
async findAll(query) {
  const where: FindOptionsWhere<Task> = {};        // typed, not Record<…>
  if (query.status) where.status = query.status;
  if (query.search) where.title = ILike(`%${query.search}%`); // case-insensitive

  const [items, total] = await this.repo.findAndCount({
    where, order: { id: 'ASC' },
    skip: (query.page - 1) * query.limit,    // offset
    take: query.limit,                       // page size
  });
  return { items, total, page: query.page, limit: query.limit };
}

// DELETE — one round-trip, "affected" gives us the 404 without a prior SELECT
async remove(id) {
  const result = await this.repo.delete(id);
  if (result.affected === 0) throw new NotFoundException(`Task ${id} not found`);
}
findAndCount returns [rows, total]; the count ignores skip/take — perfect for pagination. ILike = Postgres case-insensitive LIKE; %…% makes it "contains". delete().affected tells us if a row matched, so a missing id still 404s.

Seeding on boot — with a caveat

To keep the dev experience friendly, onModuleInit() seeds two tasks only if the table is empty. That's idempotent across a single-instance restart (we proved it: restarts don't duplicate the seed). The honest caveat, now noted in the code: it's a check-then-act, so two instances booting against the same empty DB could both seed. Harmless here; worth knowing.

07Migrations & the schema workflow

Versioned, reviewable, deliberate schema changes.

A migration is a timestamped file with an up() (apply) and down() (undo). We generated the first one by diffing the entity against the empty database:

// src/database/migrations/…-InitTasks.ts  (generated, then committed)
async up(q) {
  await q.query(`CREATE TYPE "tasks_status_enum" AS ENUM('OPEN','IN_PROGRESS','DONE')`);
  await q.query(`CREATE TABLE "tasks" ( "id" SERIAL …, "status" "tasks_status_enum"
            NOT NULL DEFAULT 'OPEN', "createdAt" TIMESTAMP NOT NULL DEFAULT now(), … )`);
}
async down(q) {                       // the exact reverse
  await q.query(`DROP TABLE "tasks"`);
  await q.query(`DROP TYPE "tasks_status_enum"`);
}
The loop to memorize

Whenever you change an entity: pnpm migration:generate ./src/database/migrations/Name → read the generated SQL → pnpm migration:run. A plain app restart will not add your new column, because synchronize is false. That's a feature, not a bug — no surprise schema changes on deploy.

TypeORM tracks which migrations have run in a bookkeeping table it auto-creates (named migrations). On migration:run it wraps everything in a transaction, applies the new ones, and records them — so running it twice is safe (the second time, "0 migrations to execute").

08Docker & environment

Making the project self-contained.

A docker-compose.yml brings up Postgres 16 with a named volume (so data survives docker compose down) and a healthcheck. The DB credentials in compose match the DB_* vars in .env. Secrets hygiene was verified: .env is gitignored and was never committed; only .env.example (dev placeholders) is tracked.

09Case study: the crash on 404 fixed

The single most valuable lesson of this change.

After wiring everything up, DELETE /tasks/9999 (a missing id) didn't return 404 — it crashed the entire Node process, and the next request got a connection-refused. The logs showed the NotFoundException was thrown correctly… and then killed the app as an unhandled promise rejection.

The culprit was in the controller, and it had been lurking harmlessly the whole time:

Before — silently broken

@Delete(':id')
@HttpCode(204)
remove(@Param('id') id) {
  this.service.remove(id);  // ⚠ promise not returned
}

After — one word

@Delete(':id')
@HttpCode(204)
remove(@Param('id') id) {
  return this.service.remove(id); // ✓ Nest awaits it
}
Why it only broke now

When remove() was synchronous, it threw inside the handler and Nest caught it → clean 404. Once it became async, calling it without return produced a floating promise: the handler returned undefined immediately (Nest sent 204), and the promise rejected later with nobody awaiting it. Node 22 treats an unhandled rejection as fatal. Returning the promise lets Nest await it and route the rejection through the exception filter → 404.

The general rule

The moment a function becomes async, every caller must return or await its promise. A successful path hides the bug (the promise just resolves); only the rejecting path — here, a 404 — exposes it. This is exactly the kind of latent issue that automated end-to-end testing catches and eyeballing the diff does not.

10Concepts & gotchas worth keeping

Is the search SQL-injectable? No — and we proved why.

ILike(`%${query.search}%`) looks like string concatenation into SQL. It isn't. We traced TypeORM's source: ILike produces a FindOperator whose value is bound as a driver parameter ($1), never interpolated into the SQL text. A payload like %'; DROP TABLE tasks;-- is treated as a literal search string. (The only user-controllable-but-literal characters are the %/_ LIKE wildcards, which affect match scope, not query structure.)

Type your where clause

We changed const where: Record<string, unknown> to FindOptionsWhere<Task>. The loose type compiled fine but would happily accept a typo like where.titel. The entity-aware type turns that into a compile error — fitting for a codebase that prizes type-safety.

Graceful shutdown

We added app.enableShutdownHooks() in main.ts. Without it, a container's SIGTERM kills the process before TypeORM can run its teardown and close the connection pool. With it, in-flight work can drain and connections close cleanly — a real deployment detail.

The 204 + envelope curiosity

The global TransformInterceptor wraps responses in { data: … }, so a void delete technically becomes { data: undefined }. Harmless: Express strips the body on a 204, and undefined wouldn't serialize anyway. Noted for awareness, not a fix.

11Testing: mock vs real DB

Two valid strategies, one used at each layer.

Unit tests — mock the repository

tasks.service.spec.ts provides a tiny in-memory fake bound to getRepositoryToken(Task). No database needed; tests are fast and isolated, exercising the service's real logic against a stand-in repo.

providers: [
  TasksService,
  { provide: getRepositoryToken(Task),
    useValue: createRepositoryMock() },
]

E2E test — real app, real DB

test/app.e2e-spec.ts boots the whole AppModule, so it connects to actual Postgres. It passes today because it only hits GET /. The honest next step (see below) is a separate test database so e2e writes never touch your dev data.

Mind the async rewrite in tests

Because the service is now async, the unit tests switched from expect(() => …).toThrow() to await expect(…).rejects.toThrow(). The mock's create() injects the default status, which is why the "defaults to OPEN" test passes against the fake — remember the real default comes from Postgres, so only the e2e path truly proves it.

12What the review confirmed (& debunked)

An adversarial multi-agent review read every changed file across correctness, TypeORM/Nest practice, security, and completeness — then independently verified each finding. 20 confirmed (almost all teaching notes), 6 rejected as false positives.

Acted on

FindingStatus
DELETE handler leaked a floating promise (process crash on 404)fixed
where typed as loose Record instead of FindOptionsWhere<Task>fixed
Comment wrongly credited create() with the status defaultfixed
No graceful shutdown — pool not closed on SIGTERMfixed
Seeding's single-instance assumption was unstatednoted in code

Confirmed as deliberate / safe info

  • ILike search is injection-safe — TypeORM parameterizes the value (verified against source).
  • Double dotenv load is benign — dotenv won't clobber existing keys; both paths agree.
  • Migrations run manually, not on boot — correct, avoids surprise schema changes / multi-instance races.
  • Secrets hygiene is correct — .env gitignored & never committed; only .env.example tracked.
  • Default postgres/postgres creds — fine for local Docker; would need real secrets & not exposing port 5432 in production.

Myths the verifiers debunked rejected

  • "PATCH can't clear description" — false. @IsOptional() skips validation when the value is null, so {"description": null} validates and writes NULL.
  • "Pagination secretly depends on transform:true" — if transform were off, @IsInt would reject the string with a 400, not let garbage through.
  • "The entities/migrations asymmetry is unexplained" — the comments already explain it.

Pre-existing notes (outside this change's scope) low

The review also flagged the api-key.guard.ts (which predates this work): an unset API_KEY makes expected = '', so a blank x-api-key: header would authenticate (fail-open). It's dormant because .env sets a key, but the robust pattern is to fail closed — refuse to boot, or reject, when no key is configured. Left untouched to keep this change focused on the database.

14Where to go next

The natural follow-on lessons, in roughly increasing depth.

TopicThe lesson
Test database isolationA .env.test with its own DB_NAME so e2e writes never touch dev data; truncate/migrate between runs.
Response DTOsYou have request DTOs but return the entity directly. Add a response shape (ClassSerializerInterceptor + @Exclude) so adding an internal column never leaks it. Entities model the table; DTOs model the wire.
Transactions & concurrencyupdate() is a read-modify-write across two round-trips; two concurrent PATCHes can clobber each other. The textbook motivation for dataSource.transaction() and @VersionColumn optimistic locking.
IndexesYou filter on status and search title. A B-tree @Index() helps equality on status — but not a leading-wildcard ILike '%x%', which needs a Postgres pg_trgm GIN index. Measure first.
Config validationGive ConfigModule a Joi/zod schema so the app fails fast on a missing or malformed env var instead of silently using a default.
Production connectionTLS (ssl) and pool sizing (extra: { max }) for managed Postgres, gated behind NODE_ENV.
