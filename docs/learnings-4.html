01Why validate the environment

Fail at boot, not on the first request.

Without validation, a missing JWT_SECRET doesn't bite until someone tries to log in — in production, at the worst possible time. With a schema, the app checks every variable at startup and refuses to boot with a precise message. The earlier the failure, the cheaper the fix. This is the same "validate at the boundary" instinct as the DTOs from Part 1 — just applied to config instead of request bodies.

02The schema

src/config/env.validation.ts — a zod schema. (zod isn't Joi-shaped, so it's wired via a validate function — see next section.)

export const envSchema = z.object({
  NODE_ENV: z.enum(['development','test','production']).default('development'),
  PORT: z.coerce.number().int().positive().default(3000),
  API_KEY: z.string().min(1),                  // required by default in zod
  DB_HOST: z.string().min(1),
  DB_PORT: z.coerce.number().int().positive().default(5432), // "5432" -> 5432
  DB_USERNAME: z.string().min(1),
  DB_PASSWORD: z.string().min(1),
  DB_NAME: z.string().min(1),
  JWT_SECRET: z.string().min(16),               // short secrets are a real weakness
  JWT_EXPIRES_IN: z.string().default('1h'),
  ADMIN_EMAIL: z.email().optional(),
  ADMIN_PASSWORD: z.string().min(8).optional(),
  REDIS_HOST: z.string().min(1).default('localhost'),
  REDIS_PORT: z.coerce.number().int().positive().default(6379),
});

export type EnvVars = z.infer<typeof envSchema>;   // the type, for free

The schema does double duty: fields are required by default (no .optional()), .default() fills fallbacks, and z.coerce.number() coerces the string env values into real numbers. Best of all, z.infer derives the EnvVars type from the schema — so the type can never drift from the rules.

03Wiring it into ConfigModule

validationSchema is the Joi-only option. For zod we pass a validate function instead — it receives the raw env, and whatever it returns becomes the validated config (or it throws to abort the boot):

// env.validation.ts
export function validateEnv(config: Record<string, unknown>): EnvVars {
  const result = envSchema.safeParse(config);   // collects ALL issues, not just the first
  if (!result.success) {
    const issues = result.error.issues
      .map((i) => `${i.path.join('.')}: ${i.message}`).join('; ');
    throw new Error(`Config validation error: ${issues}`);
  }
  return result.data;                            // coerced + defaulted
}

// app.module.ts
ConfigModule.forRoot({ isGlobal: true, validate: validateEnv }),

04Fail-fast, demonstrated verified

Booting with a bad variable now stops the app cold with an exact message:

$ DB_PORT=notaport node dist/main.js
ERROR [ExceptionHandler] Error: Config validation error: DB_PORT: Invalid input: expected number, received NaN

$ JWT_SECRET=tiny node dist/main.js
ERROR [ExceptionHandler] Error: Config validation error: JWT_SECRET: Too small: expected string to have >=16 characters

With a valid .env, the app boots normally — and the 5 schema unit tests cover defaults, coercion, a missing required var, a too-short secret, and the report-all-errors behavior.

05Gotchas

  • The migration CLI isn't covered. data-source.ts reads process.env directly (it runs outside Nest), so the schema only guards the app boot, not migration:run. A fair trade — the app is where bad config does the damage.
  • zod gives you the type for free. type EnvVars = z.infer<typeof envSchema> — no separate interface to keep in sync, unlike the cast you'd write with Joi.
  • zod v4 nuance: the email check is z.email() (top-level), not the deprecated z.string().email(); and numeric coercion is z.coerce.number().
  • Defaults are applied to the validated config, so ConfigService.get('PORT') returns the coerced/defaulted value, not the raw string.
