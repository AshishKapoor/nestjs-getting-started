01Why generated docs

The spec that can't go stale.

Hand-written API docs drift from the code the moment someone forgets to update them. Swagger flips that: the docs are derived from the code — routes, params, DTOs, validators — so they're always in sync. The output is OpenAPI, a standard machine-readable contract you can hand to client-codegen tools, Postman, or a frontend team.

02The CLI plugin does the heavy lifting

One line in nest-cli.json and your DTOs become schemas.

"compilerOptions": {
  "deleteOutDir": true,
  "plugins": ["@nestjs/swagger"]   // auto-adds @ApiProperty from TS types + validators
}

Without the plugin you'd decorate every DTO field with @ApiProperty() by hand. With it, the compiler inspects each property's TypeScript type and its class-validator decorators and emits the schema for you. That's why CreateTaskDto shows up in the spec with title marked required (from @MinLength) and description/status optional (from @IsOptional) — without touching the DTO.

03Setup in main.ts

const config = new DocumentBuilder()
  .setTitle('Tasks API')
  .setDescription('NestJS getting-started API …')
  .setVersion('1.0')
  .addBearerAuth({ type:'http', scheme:'bearer', bearerFormat:'JWT' }, 'jwt')
  .addApiKey({ type:'apiKey', name:'x-api-key', in:'header' }, 'api-key')
  .build();
const document = SwaggerModule.createDocument(app, config);
SwaggerModule.setup('docs', app, document);   // UI at /docs, raw spec at /docs-json

04Documenting auth

The two security schemes registered above are wired to routes with tiny decorators, so the Swagger UI's "Authorize" button knows what each endpoint needs:

DecoratorOnEffect
@ApiTags('tasks')controllersgroups endpoints in the UI
@ApiBearerAuth('jwt')/auth/me, /auth/adminmarks them as needing a JWT
@ApiSecurity('api-key')POST/PATCH/DELETE /tasksmarks them as needing x-api-key

05Verified spec live

Fetching /docs-json from the running app confirms the generated contract:

title: Tasks API 1.0
paths: [/, /auth/admin, /auth/login, /auth/me, /auth/register, /tasks, /tasks/{id}]
securitySchemes: [api-key, jwt]
schemas: [CreateTaskDto, UpdateTaskDto, RegisterDto, LoginDto]
CreateTaskDto props: [title, description, status]  required: [title]
/auth/me   security: [{ jwt: [] }]
POST /tasks security: [{ api-key: [] }]

And GET /docs serves the interactive Swagger UI (HTTP 200).

06Gotchas

  • The plugin only runs through the Nest compiler (nest build/nest start). Tests run via ts-jest and don't get the auto-@ApiProperty — fine, since tests don't assert on the spec.
  • Response shapes aren't inferred when handlers return loosely-typed values. For precise response docs, add @ApiOkResponse({ type: ... }) — left out here to keep the diff small.
  • Swagger reflects your declared security, not your actual guards. The decorators and the @UseGuards must agree by hand — Swagger won't catch a mismatch.
  • Don't expose /docs unauthenticated in production unless you intend the API surface to be public.
