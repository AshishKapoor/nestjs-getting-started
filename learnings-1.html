01The mental model

You describe what with decorators; Nest wires the how.

NestJS is an opinionated Node framework with an Angular-style architecture (modules, dependency injection, decorators) sitting on top of Express. It's TypeScript-first. The core trade you're making: instead of manually constructing and passing objects around, you declare your pieces with decorators and let Nest's container assemble the whole graph for you.

The one sentence to remember

A Module groups a feature and defines its DI scope; a Controller maps HTTP routes to methods and stays thin; a Provider/Service holds the logic and is injected, never new-ed. Everything else is a cross-cutting tool that hooks into the request as it flows through.

02The 3 building blocks

95% of what you write is one of these three.

BlockDecoratorJob
Module@Module()Groups a feature; defines its DI scope. Root is AppModule.
Controller@Controller('x')Maps HTTP routes → methods. Stays thin — delegates to services.
Provider / Service@Injectable()Business logic. Injected, never instantiated by hand. exports to share.

Module

@Module({
  controllers: [TasksController],
  providers: [TasksService],
  // exports: [TasksService]
})
export class TasksModule {}

Controller

@Controller('tasks')
export class TasksController {
  @Get()
  findAll(@Query() q) {
    return this.svc.findAll(q);
  }
}

Service

@Injectable()
export class TasksService {
  findAll(q) {
    /* logic here */
  }
}

A provider is only injectable where it's in scope: listed in the providers of an imported module, or exports-ed by one.

03Dependency injection — the heart of Nest

Declare a dependency in a constructor; Nest builds it once and hands it to you.

You never write new TasksService(). You ask for it in the constructor and the container provides a single shared instance (default scope = singleton). This keeps controllers thin and makes everything trivially testable — you can swap any dependency for a fake (see Testing).

@Controller('tasks')
export class TasksController {
  // injected automatically — the DI container resolves it
  constructor(private readonly tasksService: TasksService) {}
}

You can see real DI in this project's guard, which asks for two framework providers at once:

// src/common/guards/api-key.guard.ts
constructor(
  private readonly reflector: Reflector,      // reads route metadata
  private readonly config: ConfigService,     // reads env vars
) {}

04The request lifecycle order matters

Every request flows through the same ordered pipeline. Memorize this.

MiddlewareExpress-style; logging, request-id
→
Guardscan it proceed? auth → 401/403
→
Interceptor (pre)wrap call; start timer
→
Pipesvalidate + transform → 400
→
Handleryour controller → service
→
Interceptor (post)reshape response, e.g. {data}
→
Exception filteronly if something threw
The consequence that bites people

If a guard or pipe throws, the handler and the post-interceptors never run — Nest jumps straight to the exception filter. That's exactly how a failed API key (401) or invalid body (400) short-circuits before any business logic executes. (Part 2 shows the flip side: a handler that returns too early can let an async error escape the pipeline entirely.)

The other 5% — each hooks into a specific point of that pipeline. All six exist in this project.

Middleware — runs first, knows nothing about the handler

@Injectable()
export class LoggerMiddleware implements NestMiddleware {
  private readonly logger = new Logger('Request');
  use(req, _res, next) {
    this.logger.log(`-> ${req.method} ${req.originalUrl}`);
    next();                       // MUST call next() to continue
  }
}

Guard — "can this proceed?" + the custom decorators it reads

The guard reads @Public() metadata via the Reflector to decide whether to skip auth, and on success attaches a fake user to the request for downstream @User() to read.

// api-key.guard.ts
canActivate(context: ExecutionContext): boolean {
  const isPublic = this.reflector.getAllAndOverride(IS_PUBLIC_KEY, [
    context.getHandler(), context.getClass(),   // method, then controller
  ]);
  if (isPublic) return true;                // @Public() routes skip auth

  const apiKey = request.header('x-api-key');
  if (apiKey !== expected)
    throw new UnauthorizedException('Invalid or missing x-api-key header');

  request.user = { name: 'Ashish Kapoor' };   // for @User() downstream
  return true;
}
// public.decorator.ts — SetMetadata factory
export const IS_PUBLIC_KEY = 'isPublic';
export const Public = () => SetMetadata(IS_PUBLIC_KEY, true);

// user.decorator.ts — createParamDecorator, pulls req.user (or one field)
export const User = createParamDecorator(
  (data, ctx) => {
    const user = ctx.switchToHttp().getRequest().user;
    return data ? user?.[data] : user;   // @User('name') vs @User()
  },
);

Interceptors — wrap the handler with RxJS

next.handle() returns an Observable of the handler's return value. tap() runs a side effect without changing it (timing); map() transforms it (the response envelope).

Timing (global)

const start = Date.now();
return next.handle().pipe(
  tap(() => this.logger.log(
    `${method} ${url} +${Date.now()-start}ms`)),
);

Envelope (on TasksController)

return next.handle().pipe(
  map((data) => ({ data })),
);
// every response → { data: ... }

Pipes — validate & transform input

Two flavors here: the global ValidationPipe (validates every @Body()/@Query() against its DTO — see next section) and the built-in ParseIntPipe on a route param:

@Get(':id')
findOne(@Param('id', ParseIntPipe) id: number) {
  // "1" -> 1, and 400s automatically on non-numeric input
  return this.tasksService.findOne(id);
}

Exception filter — shape the error response

@Catch(HttpException) limits this filter to HTTP exceptions (and subclasses like NotFoundException). It turns any thrown HttpException into a consistent JSON body:

@Catch(HttpException)
export class HttpExceptionFilter implements ExceptionFilter {
  catch(exception, host) {
    const status = exception.getStatus();
    response.status(status).json({
      statusCode: status,
      path: request.url,
      timestamp: new Date().toISOString(),
      error: /* string or .message */,
    });
  }
}

This is why a 404 in this API comes back as { "statusCode": 404, "path": "/tasks/1", "timestamp": …, "error": "Task with id 1 not found" } rather than Nest's default shape.

06DTOs & validation — how you trust input

A DTO is the declared shape of an incoming request; class-validator decorators enforce it.

The global ValidationPipe reads these decorators at runtime. If validation fails, Nest auto-responds 400 before your controller runs. Three DTOs show the patterns:

// create-task.dto.ts — the base shape
export class CreateTaskDto {
  @IsString() @MinLength(3)        title: string;
  @IsString() @IsOptional()        description?: string;
  @IsEnum(TaskStatus) @IsOptional() status?: TaskStatus;
}

// update-task.dto.ts — DRY: same rules, all optional
export class UpdateTaskDto extends PartialType(CreateTaskDto) {}

// query-tasks.dto.ts — query strings arrive as strings...
export class QueryTasksDto {
  @IsOptional() @IsEnum(TaskStatus) status?: TaskStatus;
  @IsOptional() @IsString()         search?: string;
  @IsOptional() @Type(() => Number) @IsInt() @Min(1) page = 1;
  @IsOptional() @Type(() => Number) @IsInt() @Min(1) @Max(100) limit = 10;
}
PartialType derives an all-optional DTO that keeps the same validation rules — no copy-paste. @Type(() => Number) + transform:true coerces "?page=2" to a real number before @IsInt runs.

The pipe is configured once, globally, in main.ts:

app.useGlobalPipes(new ValidationPipe({
  whitelist: true,             // strip props with no decorator
  forbidNonWhitelisted: true,  // 400 if client sends unknown props
  transform: true,             // turn JSON into DTO instances + coerce types
}));

07Wiring it together

Where each tool is attached — and the scope ladder that governs it.

Scope goes method → controller → global. You can apply a tool with a decorator on a single handler, on a whole controller, or app-wide. For globals there are two routes: imperatively in main.ts (app.useGlobalX()), or via the APP_* provider tokens — which is preferred when the tool needs dependency injection.

Globals via DI tokens — app.module.ts

providers: [
  AppService,
  { provide: APP_INTERCEPTOR, useClass: LoggingInterceptor },
  { provide: APP_FILTER,      useClass: HttpExceptionFilter },
],
// ConfigModule.forRoot({ isGlobal: true }) loads .env
// middleware can't use a decorator — wired in configure():
configure(consumer) {
  consumer.apply(LoggerMiddleware).forRoutes('*');
}

Controller-scoped — tasks.controller.ts

@UseGuards(ApiKeyGuard)            // auth for this controller
@UseInterceptors(TransformInterceptor) // {data} envelope
@Controller('tasks')
export class TasksController {

  @Get() @Public()              // method-scoped: skip the guard
  findAll(@Query() q) { ... }

  @Post()
  create(@Body() dto, @User('name') by) { ... }
}
Why APP_FILTER/APP_INTERCEPTOR instead of main.ts

Registering a global through these tokens lets it participate in DI — so the guard can inject ConfigService, the filter could inject a logger, etc. The plain app.useGlobalX() form can't inject providers. That's the whole reason both styles exist.

09Testing

DI makes everything swappable; two layers of tests use that.

Unit — a tiny DI module

const mod = await Test.createTestingModule({
  controllers: [AppController],
  providers: [AppService],
}).compile();
const ctrl = mod.get(AppController);
expect(ctrl.getHello()).toBe('Hello World!');

Swap a real dependency for a mock with .overrideProvider(Dep).useValue(mock).

E2E — the real app + supertest

const app = mod.createNestApplication();
await app.init();
return request(app.getHttpServer())
  .get('/')
  .expect(200)
  .expect('Hello World!');

Boots the whole module graph and makes real HTTP requests against it.

10The Nest CLI — your daily driver

npx nest g resource tasks   # scaffolds module+controller+service+dto+tests (full CRUD)
npx nest g module orders
npx nest g controller orders
npx nest g service orders
npx nest g guard auth        # also: interceptor | pipe | filter | middleware | decorator
nest start --watch           # dev (pnpm start:dev)
nest build                   # compile to dist/

g resource is the big time-saver — it scaffolds an entire REST (or GraphQL) CRUD module in one command.

11Mental models & gotchas

  • Controllers stay thin. They translate HTTP ↔ method calls. All logic belongs in services.
  • Order is everything. Guard/pipe failures short-circuit before the handler; the exception filter is the only thing that runs after a throw.
  • Scope ladder: method @Use… on handler → controller @Use… on class → global main.ts or APP_*.
  • Use APP_* tokens for any global that needs DI; app.useGlobalX() otherwise.
  • DTOs are your trust boundary. whitelist strips unknowns, forbidNonWhitelisted rejects them, transform coerces types.
  • Metadata + Reflector is the pattern behind @Public(), @Roles(), and most declarative behavior.
