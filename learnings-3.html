01Authentication vs authorization

Two different questions, two different guards.

Authentication answers "who are you?" — proven by a signed token. Authorization answers "are you allowed to do this?" — decided by your role. Conflating them is a classic source of security bugs, so this step keeps them in separate guards: JwtAuthGuard authenticates, RolesGuard authorizes.

JWT (JSON Web Token) makes auth stateless: the server signs a small token at login, the client sends it on every request, and the server verifies the signature — no session store, no DB lookup per request.

Issue the token (login)

const ok = user &&
  await bcrypt.compare(dto.password, user.passwordHash);
if (!ok) throw new UnauthorizedException('Invalid credentials');

const payload = { sub: user.id, email: user.email, role: user.role };
return { accessToken: await this.jwt.signAsync(payload) };

Verify the token (strategy)

class JwtStrategy extends PassportStrategy(Strategy, 'jwt') {
  constructor(config) {
    super({
      jwtFromRequest: ExtractJwt.fromAuthHeaderAsBearerToken(),
      secretOrKey: config.get('JWT_SECRET'),
    });
  }
  validate(payload) {        // only runs if signature is valid
    return { userId: payload.sub, email: payload.email, role: payload.role };
  }
}
JwtModule.registerAsync — secret from config, not hard-coded

The signing secret comes from JWT_SECRET via ConfigService (the registerAsync + useFactory + inject pattern). Part 4 adds a schema so the app refuses to boot if that secret is missing.

04Two guards, in order

On the admin route, both guards run — authentication first, then authorization. Order matters: RolesGuard relies on req.user that JwtAuthGuard populated.

@UseGuards(JwtAuthGuard, RolesGuard)   // authn, then authz
@Roles(UserRole.ADMIN)
@Get('admin')
adminOnly(@CurrentUser('email') email: string) {
  return { message: `Welcome, admin ${email}` };
}

A request with no token → 401 (JwtAuthGuard). A valid USER token → 403 (RolesGuard). An ADMIN token → 200. The @Roles()/Reflector mechanism is the exact same metadata pattern as @Public() from Part 1.

05Security decisions that matter

  • No role in RegisterDto. Accepting a client-chosen role would let anyone register as ADMIN — a privilege-escalation hole. New users always default to USER.
  • Identical login error. Wrong email and wrong password both return "Invalid credentials" / 401, so an attacker can't enumerate which emails exist.
  • The hash never leaves the server. register returns a SafeUser with passwordHash stripped; it's never serialized.
  • Passwords are bcrypt-hashed (cost 10) — never stored or logged in plaintext.
  • Secret lives in config, not source. The dev seed admin is gated behind env vars and flagged dev-only.
Relationship to the existing ApiKeyGuard

Part 1's ApiKeyGuard still protects the Tasks routes — a single shared key that proves "authenticated" but not "who." JWT is the per-user upgrade. They coexist here for teaching; a real app would pick one and, to put Tasks behind JWT, apply JwtAuthGuard globally (it already honors @Public()) and read the owner from req.user.

07Gotchas

  • import type for decorated params. With isolatedModules + emitDecoratorMetadata, a pure type used as a @CurrentUser() user: JwtUser param must be imported as type, or the build fails (TS1272).
  • expiresIn typing. @nestjs/jwt types it via the ms library's StringValue union, so a plain config string needs a cast to the option's own type.
  • JWT is stateless. You can't "log out" a token server-side without extra machinery (a denylist or short expiry + refresh tokens). That's the trade for not having sessions.
  • Don't put secrets in the payload. A JWT is signed, not encrypted — anyone can base64-decode and read the claims. Keep it to ids and roles.

08Where this leads

This auth works, but a production system would add: re-loading the user in validate() (to reject deleted/disabled accounts), refresh tokens for longer sessions, and moving the Tasks routes behind JWT with per-user ownership. First, though — Part 4 hardens the thing all of this depends on: the environment config, with a schema that fails fast when JWT_SECRET (or any required var) is missing.
