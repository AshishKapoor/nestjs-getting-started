01Three primitives, one theme

Keep slow or repeated work out of the user's request.

All three tools share a goal: don't make the HTTP request wait for work it doesn't need to. A cron job runs on a timer with no request at all; a cache skips repeated expensive work; a queue defers slow side effects to a background worker so the response returns immediately. This step adds a representative slice — the microservices/gRPC transport is deliberately out of scope for this REST app.

⏱ Schedule

Run code on a timer. No request involved.

⚡ Cache

Reuse a result for a TTL window instead of recomputing.

📬 Queue

Hand work to a worker; respond now, process later.

02Scheduling with @Cron

tasks/tasks.scheduler.ts

@Injectable()
export class TasksScheduler {
  constructor(private readonly stats: TasksStatsService) {}

  @Cron('*/5 * * * * *')   // 6 fields: sec min hour dom mon dow → every 5s
  async heartbeat() {
    const { total } = await this.stats.getStats();
    this.logger.log(`heartbeat: ${total} task(s) in the DB`);
  }
}

Enabled by ScheduleModule.forRoot() in AppModule, which scans providers for @Cron/ @Interval/@Timeout at startup. Use CronExpression constants for common schedules.

03Caching

tasks/tasks-stats.service.ts — manual get/set around an "expensive" DB count.

async getStats() {
  const hit = await this.cache.get<TaskStats>(KEY);
  if (hit) return { ...hit, cached: true };       // HIT — no DB
  const total = await this.tasks.count();           // MISS — query
  const stats = { total, generatedAt: new Date().toISOString(), cached: false };
  await this.cache.set(KEY, stats, 15_000);          // TTL in ms (cache-manager v7)
  return stats;
}

Because generatedAt is fixed when the value is computed, two calls within the 15s window return the same timestamp — visible proof the second call never touched the database. CACHE_MANAGER comes from a global CacheModule.register({ isGlobal: true }).

04Queues with Bull + Redis

A producer enqueues; a consumer processes — connected through Redis.

→
Redisholds the job

RemindersProcessor@Process('remind') runs later
// producer — returns immediately
@Injectable() class RemindersService {
  constructor(@InjectQueue('reminders') private queue: Queue) {}
  enqueue(taskId) { return this.queue.add('remind', { taskId }); }
}

// consumer — runs in the background, after the response is sent
@Processor('reminders') class RemindersProcessor {
  @Process('remind')
  handle(job: Job<{ taskId: number }>) {
    this.logger.log(`processing reminder job ${job.id} for task ${job.data.taskId}`);
  }
}

06Verified live from the logs

[TasksStats] stats cache MISS — querying DB        ← first call hits the DB
[TasksStats] stats cache HIT                       ← subsequent calls don't
[TasksCron]  heartbeat: 2 task(s) in the DB        ← fires every 5s, no request
[TasksCron]  heartbeat: 2 task(s) in the DB
[Reminders]  processing reminder job 1 for task 3  ← ran AFTER POST returned 201

And GET /tasks/stats returns the same generatedAt on back-to-back calls — the cache in action.

07Gotchas

  • Route order: @Get('stats') must precede @Get(':id') — otherwise "stats" matches the :id param and ParseIntPipe 400s on it.
  • cache-manager v7 ttl is milliseconds. Older majors used seconds — an easy off-by-1000 trap.
  • import type again. Queue, Job, and Cache are pure types used in decorated constructor/handler signatures, so they must be type-imported (TS1272).
  • The processor runs in the same process here. Real scaling moves consumers to separate worker processes so background load doesn't compete with request handling.
  • Bull needs Redis to boot. The app (and e2e tests) now depend on Redis being up, just as they depend on Postgres. Both are in docker compose up -d.
  • Out of scope: microservices/gRPC transports — they don't fit this single REST service, so they were intentionally left for a dedicated project.
