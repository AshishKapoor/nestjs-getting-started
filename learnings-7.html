01Why lifecycle hooks

Resources need a clean open and a clean close.

Connections, pools, queues, timers, and caches all need to be set up after the app is wired and torn down before it exits. Nest exposes named hooks for each phase so you can put that logic in the right place — and, crucially, run teardown before the process dies so in-flight work finishes and external systems aren't left with dangling connections. You've already used these implicitly: the DB seeding (Part 2), the admin seed (Part 3), and the enableShutdownHooks() that lets TypeORM and Bull clean up.

02The order, verified from the logs

A single provider implementing all five hooks revealed the exact sequence at runtime — startup top-down, then a SIGTERM triggering teardown:

  1. ↑1
    onModuleInit()
    this module's dependencies are ready — open connections, seed data
  2. ↑2
    onApplicationBootstrap()
    ALL modules initialized — the app is about to start listening
  3. ↓1
    onModuleDestroy()
    release this module's resources
  4. ↓2
    beforeApplicationShutdown(signal)
    last chance before teardown; receives the signal name
  5. ↓3
    onApplicationShutdown(signal)
    app is closing — pools/queues get closed here
[Lifecycle] startup 1/2 · onModuleInit — module deps ready
[Lifecycle] startup 2/2 · onApplicationBootstrap — all modules ready, about to listen
… kill -TERM <pid> …
[Lifecycle] shutdown 1/3 · onModuleDestroy — releasing module resources
[Lifecycle] shutdown 2/3 · beforeApplicationShutdown (signal=SIGTERM)
[Lifecycle] shutdown 3/3 · onApplicationShutdown (signal=SIGTERM) — up ~0s, resources closed

03One provider, every hook

src/common/lifecycle/lifecycle.service.ts

@Injectable()
export class LifecycleService
  implements OnModuleInit, OnApplicationBootstrap,
    OnModuleDestroy, BeforeApplicationShutdown, OnApplicationShutdown {

  onApplicationShutdown(signal?: string) {
    this.logger.log(`onApplicationShutdown (signal=${signal})`);  // e.g. SIGTERM
  }
  // …the other four hooks log their phase too
}

In real code you implement only the hooks you need — a provider that opens a gRPC channel implements onModuleInit + onModuleDestroy; one that flushes metrics implements onApplicationShutdown. Implementing an interface is all it takes; Nest discovers and calls it.

04What graceful shutdown buys you

The one line that makes it work

app.enableShutdownHooks() in main.ts (added back in Part 2) registers the SIGTERM/SIGINT listeners. Without it, the shutdown hooks never fire — the process just dies. With it, a container told to stop (a Kubernetes rollout, a docker stop) gives the app a moment to:

SubsystemWhat its shutdown hook does
TypeORM (Part 2)DataSource.destroy() — closes the Postgres connection pool
Bull (Part 6)closes the queue's Redis connections; lets in-flight jobs settle
Your providersflush buffers, deregister from service discovery, close sockets

05Gotchas

  • Hooks are inert without enableShutdownHooks() — and it's opt-in because attaching process signal listeners has a small cost Nest doesn't impose by default.
  • Shutdown hooks add latency to exit. Keep them fast and don't start new work in them; a stuck hook delays the process from terminating.
  • Startup is bottom-up, shutdown unwinds it. Don't assume another module's resource is still open in your onModuleDestroy — order across modules isn't something to rely on for correctness.
  • Async hooks are awaited. Return a promise and Nest waits for it — handy for genuinely async teardown.
